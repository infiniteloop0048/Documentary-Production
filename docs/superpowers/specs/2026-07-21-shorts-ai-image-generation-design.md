# Shorts: AI-Generated Images Footage Source — Design Spec

**Date:** 2026-07-21
**Status:** Approved
**Scope:** Add a third footage-source option to the Short/Reel pipeline: `"ai_image"` —
per-scene still images generated by a text-to-image model (not searched from stock
libraries), animated with the existing Ken Burns pipeline. Sits alongside the existing
`"video"` and `"image"` (stock photo) sources added in the prior session. The rest of the
pipeline — TTS, word-timing, sentence/segment timing, captions, music, beat-sync,
cinematic ending, concat/mux — is unchanged and identical across all three sources.

## Standing constraints (from CLAUDE.md, unchanged)

- Do not touch `pipeline/`, top-level `runner/`, existing files in `adapters/` (except
  adding new ones), `history/`, `licensing.py`, or existing test files unless fixing a
  genuine bug.
- Correct venv is `.venv/`, never `venv/`.
- Bump the cache-busting `?v=N` query string in `docu_studio/gui/webview_app.py` after any
  HTML/JS edit — current value `?v=16`; this change bumps it to `?v=17`.
- No real, paid API calls during implementation/testing without the user's explicit
  go-ahead at that point — all new adapters are tested with mocked HTTP, matching the
  existing `test_pexels_photos_adapter.py`-style convention.

## Core principle: same timing architecture, AI-generated instead of searched stills

Exactly the same audio-first, scene-by-scene timing as video/stock-image mode: one
continuous TTS voiceover, word-level timestamps, `sentence_spans` + `plan_cuts` (both
untouched) drive segment durations. The only new thing is **where each segment's still
image comes from** and **how detailed/story-aware its prompt is**:

- **Stock-image mode (existing, unchanged):** a per-sentence pool of *searched* stock
  photos, picked by a short keyword query.
- **AI-image mode (new):** one *generated* image per sentence, produced from a detailed,
  scene-specific prompt written by the LLM — not searched, created to match the narration
  exactly.

**Key finding: rendering needs zero new code.** `_build_segment()`'s existing
`kind == "image"` branch (added in the prior session) calls `apply_ken_burns_image` on
whatever local file path is in the pool entry — it has no idea, and doesn't need to know,
whether that file was downloaded from Pexels or generated by GPT-Image-1. AI-generated
images reuse that branch verbatim by tagging their pool entries `"kind": "image"` too. All
new logic is in **prompt generation** (`shorts_script_gen.py`) and **collection**
(a new function in `shorts_assembly.py`), not rendering.

## Provider landscape (researched, not assumed)

Veo and Kling, mentioned in the original request, are **video**-generation models (they
produce short video clips, not stills) — not a fit for a still-image-per-scene pipeline,
so they're excluded. "Nano Banana" is real and does fit: it's Google's public nickname for
its Gemini image-generation model family. Four providers were selected (multi-select
answer): OpenAI, Google Gemini, Replicate, fal.ai — spanning premium single-vendor APIs
and multi-model aggregators, i.e. cheap to costly as requested.

| Preset id | Provider | Model | Tier |
|---|---|---|---|
| `openai_gpt_image_1` | OpenAI | GPT-Image-1 | Premium |
| `gemini_nano_banana` | Google Gemini | Gemini 2.5 Flash Image ("Nano Banana") | Mid |
| `gemini_nano_banana_pro` | Google Gemini | Gemini 3 Pro Image ("Nano Banana Pro") | Premium |
| `replicate_flux_schnell` | Replicate | `black-forest-labs/flux-schnell` | Cheap |
| `replicate_flux_dev` | Replicate | `black-forest-labs/flux-dev` | Mid |
| `replicate_sdxl` | Replicate | `stability-ai/sdxl` | Cheap |
| `fal_flux_schnell` | fal.ai | `fal-ai/flux/schnell` | Cheap |
| `fal_flux_dev` | fal.ai | `fal-ai/flux/dev` | Mid |

Each preset carries a hardcoded published per-image price (from each vendor's public
pricing page at spec time) used only for the GUI's pre-run cost estimate — no live pricing
API exists for any of these, so this is a static table, refreshed manually if prices
change.

New credentials needed: `docu_studio_gemini`, `docu_studio_replicate`, `docu_studio_fal`
(new keyring entries). OpenAI reuses the existing `docu_studio_openai` key already used for
LLM calls — same account, different endpoint (`images.generate` vs `chat.completions`).

## Investigation findings

- **`ImageGenProvider` is a new ABC shape, not a reuse of `PhotoProvider`.**
  `PhotoProvider.search(query, page) -> list[PhotoResult]` returns a *pool of candidates* to
  dedupe/choose from — meaningful for a search API, meaningless for text-to-image (one
  prompt in, one image out, no candidates). The new ABC is
  `ImageGenProvider.generate(prompt: str, width: int, height: int, model: str) -> bytes` —
  raw image bytes, so the adapter itself absorbs each vendor's response-shape differences
  (OpenAI/Gemini return base64 payloads; Replicate/fal.ai return a hosted URL that must be
  downloaded) behind one uniform return type. Callers write those bytes to
  `scene_dir / f"ai_img_{i:03d}.png"` themselves — no separate "download" step needed for
  the base64 providers, and a single shared `_download_bytes(url)` helper covers the
  URL-returning ones.
- **`retry.py`'s `@retry` decorator is provider-agnostic** (catches `Exception`, exponential
  backoff, re-raises after exhaustion) and is already used by every existing footage/photo
  adapter (`pexels_photos_adapter.py` confirmed). Every new `ImageGenProvider.generate()`
  implementation gets the same `@retry(max_attempts=3, ...)` treatment — no new retry
  primitive needed.
- **`shorts_script_gen.py`'s existing per-sentence JSON call is the extension point for
  detailed prompts.** `_fetch_scene_json()` / `_QUERY_OVERRIDE_TEMPLATE` already asks the
  LLM for one JSON entry per sentence with a `title` (short stock-search query) and (on the
  first entry) `music_moods`. A new template variant adds an `image_prompt` field to every
  entry: a full descriptive text-to-image prompt (subject, action, setting, camera framing,
  lighting, mood) built from that sentence's narration. The existing `title` field is kept
  on every entry regardless of mode — it's the fallback-to-stock-search query needed for the
  AI-generation-failure path.
- **Story continuity needs one extra upfront LLM call, not a bigger per-sentence call.**
  When `ai_story_continuity=True`, a new `_generate_style_guide(llm, script_text) -> str`
  call runs once before the per-sentence call, asking for a short reusable description (art
  style/medium, color palette, recurring subject/character appearance, setting). That text
  is substituted into the per-sentence template's instructions ("every `image_prompt` must
  open with this style/character description, then add this scene's specific action") so
  the model repeats it consistently across every entry in the *same* call, rather than
  relying on the model to invent a consistent style unprompted. When
  `ai_story_continuity=False`, this call is skipped and the per-sentence template omits the
  style-guide instruction — each `image_prompt` is generated from its own sentence alone,
  matching the "independent, image per scene" behavior the user asked to keep available.
- **Collection is structurally different from both existing collectors, not a third copy of
  the same pattern.** `_collect_clips_per_sentence`/`_collect_images_per_sentence` build a
  *pool of candidates per sentence* (many results, deduped) because search can return
  several matches. AI generation produces exactly one image per prompt — there's no pool to
  build. The new `_collect_ai_images_per_sentence()` instead, per sentence: calls
  `image_gen_provider.generate(prompt=script.image_prompts[i], ...)`; on success, writes the
  bytes to disk and yields a single-entry pool `[{"path", "width", "height",
  "kind": "image"}]`; on failure (after the adapter's own retries are exhausted), falls back
  to one stock-photo search using `script.visual_queries[i]` (reusing the existing
  `_search_one_photo_query` + download helpers from `_collect_images_per_sentence`) and
  emits a `LogEvent` (`level=LogLevel.WARNING`, e.g. `"AI image generation failed for scene
  3 — used stock photo instead."`) so the failure is visible to the user, per their answer.
  If *that* also fails (no stock providers configured, or stock search also empty), the
  sentence's pool is legitimately empty — handled by the same empty-pool-falls-back-to-
  `fallback_pool` resolution logic `assemble_short()` already has for stock mode. The
  `fallback_pool` itself, for AI-image mode, is built once via one stock-photo search on
  `script.visual_queries[0]` (not an extra AI generation) — reusing existing stock
  infrastructure for this rare double-failure safety net avoids spending image-generation
  budget on a fallback that's usually never used.
- **`ShortsScript` (dataclass, `shorts_script_gen.py`) gains one new field:**
  `image_prompts: tuple[str, ...] = ()`, parallel to the existing `visual_queries` — empty
  for video/stock-image mode (never populated, never read), populated only when
  `generate_shorts_script(...)` is called with the new `footage_source="ai_image"` +
  `story_continuity` parameters.
- **GUI wiring precedent (footage-source `<select>`, added in the prior session) extends
  directly** — same row-visibility toggle pattern in `app.js`, same `bridge.py`
  `config.get(...)` pattern. The new pieces (model-preset dropdown, story-continuity toggle,
  cost estimate text) are additional rows shown only when
  `footage-source-select value === "ai_image"`, following the exact same
  `style.display = ... ? '' : 'none'` toggle already used for `#aspect-row` and the new
  `#footage-source-row`.

## Changes by file

### `docu_studio/adapters/image_gen/` (new package)
- `base.py`: `ImageGenProvider(ABC)` with `generate(self, prompt: str, width: int,
  height: int, model: str) -> bytes`. No `ImageGenResult` dataclass needed — a single
  generated image has no metadata worth carrying beyond the bytes themselves (caller
  already knows the requested width/height).
- `openai_adapter.py`: `OpenAIImageGenAdapter` — calls OpenAI's image-generation endpoint,
  decodes the base64 response into bytes. `@retry`-wrapped.
- `gemini_adapter.py`: `GeminiImageGenAdapter` — calls the Gemini image-generation endpoint
  for both `gemini_nano_banana` and `gemini_nano_banana_pro` (model id passed through to
  `generate()`), decodes base64. `@retry`-wrapped.
- `replicate_adapter.py`: `ReplicateImageGenAdapter` — calls Replicate's prediction API with
  the given model id, polls for completion, downloads the returned image URL.
  `@retry`-wrapped.
- `fal_adapter.py`: `FalImageGenAdapter` — calls fal.ai's synchronous inference endpoint,
  downloads the returned image URL. `@retry`-wrapped.
- `factory.py`: `IMAGE_GEN_PRESETS: dict[str, dict]` — preset id → `{provider, model,
  price_per_image_usd, label}` (the table above). `build_image_gen_provider(preset_id: str,
  openai_key: str, gemini_key: str, replicate_key: str, fal_key: str) -> ImageGenProvider`
  looks up the preset, instantiates the right adapter class with the right key and model id.

### `docu_studio/shorts/shorts_script_gen.py`
- New `_STYLE_GUIDE_TEMPLATE` + `_generate_style_guide(llm, script_text) -> str` — one
  `generate_script`-shaped call (reuses `LLMProvider.generate_script`, no new ABC method)
  asking for a short, reusable style/character/setting description from the full script.
- New `_QUERY_OVERRIDE_TEMPLATE_AI_IMAGE` (independent mode) and
  `_QUERY_OVERRIDE_TEMPLATE_AI_IMAGE_STORY` (story mode, includes the style guide text) —
  variants of the existing `_QUERY_OVERRIDE_TEMPLATE` that additionally request an
  `image_prompt` field per entry, alongside the existing `title` field which is kept
  unconditionally.
- `ShortsScript` gains `image_prompts: tuple[str, ...] = ()`.
- `generate_shorts_script(...)` gains `footage_source: str = "video"` and
  `story_continuity: bool = True` parameters. When `footage_source == "ai_image"`: run the
  style-guide call first if `story_continuity`, then use the AI-image template variant for
  the per-sentence call and populate `image_prompts` from its `image_prompt` fields (same
  retry-once-then-fallback pattern already used for `visual_queries`/`music_moods` via
  `_fetch_scene_json`). For any other `footage_source`, behavior is byte-identical to today
  — no style-guide call, existing template, `image_prompts` stays `()`.

### `docu_studio/shorts/shorts_config.py`
- `SHORTS_FOOTAGE_SOURCES` becomes `("video", "image", "ai_image")`.
- `ShortsConfig` gains `ai_image_model: str = "replicate_flux_schnell"` (a safe cheap
  default) and `ai_story_continuity: bool = True`, both validated in `__post_init__`
  (`ai_image_model` checked against `IMAGE_GEN_PRESETS` keys when `footage_source ==

## Explicitly out of scope

- No live, paid end-to-end run during implementation — verified via mocked adapters and
  existing ffmpeg/assembly test patterns only, per the standing constraint above. A live run
  (paid, real API keys) happens only if/when the user explicitly asks for one, as with the
  stock-image feature.
- No dynamic/live pricing lookups — the cost estimate uses a static, hardcoded price table
  updated manually if vendor pricing changes.
- No Midjourney support — no official API exists to integrate against.
- No per-image resolution/quality override beyond the preset's baked-in defaults (e.g. no
  separate "draft vs. final quality" toggle) — each preset maps to one fixed
  provider+model+quality combination.
- No mid-run cost tracking/actual-spend display — only the pre-run estimate described above.
- No changes to `pipeline/`, `history/`, `licensing.py`, or existing test files beyond
  adding new test cases for this feature.
